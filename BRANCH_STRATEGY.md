# CampusHub 轻量分支策略

## 分支结构

### 1. `main`

- 只放稳定版和可演示版本
- 不直接在这里开发
- 只有确认没问题后才合并进去

### 2. `develop`

- 日常开发集成分支
- 平时大家都往这里合并
- 这个分支尽量保持能运行

### 3. `feature/*`

- 每个人开发具体功能时新建自己的分支
- 从 `develop` 拉出
- 做完后再合回 `develop`


## 分支命名

统一用这个格式：

`feature/模块-内容-姓名缩写`

示例：

- `feature/auth-login-zjh`

模块建议就按功能来写：

- `auth` 登录注册
- `profile` 个人资料
- `demand` 需求发布与浏览
- `order` 订单流程
- `credit` 评价与信用
- `message` 消息通知

## 协作流程

1. 创建任务前，先明确功能目标、负责人和验收标准。
2. 开发人员必须从 `develop` 分支创建对应的 `feature/*` 分支。
3. 功能开发完成后，应先完成自测，再提交合并请求。
4. 合并到 `develop` 前，至少应由 1 名成员完成代码检查或确认。
5. `develop` 应保持可运行、可联调；如发现冲突或阻塞问题，应优先处理。
6. 在版本稳定、满足演示或验收条件后，再将 `develop` 合并到 `main`。

## 提交文件描述
1. feat: 新功能
2. fix: 修复 Bug
3. refactor: 代码重构（无功能变化）
4. docs: 文档修改
5. style: 格式/缩进/空格（不改动逻辑）
6. test: 新增/修改测试用例
7. chore: 构建、CI、配置、依赖、工程调整
8. perf: 性能优化
9. ci: 流水线、GitHub Actions、CI 配置修改
10. build: 编译脚本、打包、依赖构建调整
11. revert: 回退某次提交
12. merge: 分支合并
13. hotfix: 紧急线上修复


## 团队约定

1. 禁止直接向 `main` 分支提交代码。
2. 每个独立任务应使用单独的 `feature` 分支开发。
3. 代码合并前必须完成自测，确保无明显功能错误。
4. 涉及核心流程、公共模块或较大范围改动的内容，合并前应至少经过 1 人确认。
