# 校园互助服务平台需求规格说明书

## 1 引言

### 1.1 目的

本文档用于正式定义“校园互助服务平台”的软件需求，作为后续系统设计、实现、测试、部署与验收的共同依据。文档在 AI 生成需求草案的基础上，结合真人问卷中反映出的真实用户需求，对功能边界、优先级和治理规则进行了补充和修订。

### 1.2 范围

本平台面向高校学生及平台管理员，提供校内互助场景下的信息发布、需求撮合、在线沟通、任务执行、评价信用、举报申诉与运营治理能力。平台重点覆盖失物招领、快递/跑腿、二手交易、学习求助、资料分享、组队招募、拼单拼车与咨询问答等校园高频场景。

### 1.3 定义

| 术语 | 说明 |
|---|---|
| 发布者 | 在平台上发起任务、咨询、交易或求助信息的用户。 |
| 响应者 | 对需求进行接单、报名、回复或协作的用户。 |
| 订单/任务记录 | 从发布、报名/接单、沟通、执行、完成到评价的全过程记录。 |
| 信用分 | 平台依据认证、履约、评价与违规情况计算的信誉指标。 |
| 站内沟通 | 用户在平台内部进行的即时消息或订单会话。 |

## 2 总体描述

### 2.1 产品视角

校园互助服务平台是一个面向校内用户的轻量级协作系统，不以开放社交为主，而以“可信身份 + 任务撮合 + 可追踪协作 + 平台治理”为核心。系统既要支持信息流场景（失物招领、资料分享、校园资讯），也要支持服务流场景（跑腿、学习辅导、二手交易、组队协作）。

### 2.2 用户特征

| 角色 | 特征描述 |
|---|---|
| 学生用户 | 主要通过移动端访问，需求高频且碎片化，重视操作简洁、信息可信、响应速度和隐私保护。 |
| 发布者 | 更关注快速发布、筛选合适对象、任务完成凭证、自动确认与纠纷处理。 |
| 响应者 | 更关注需求真实性、报酬清晰度、接单门槛、信用可见性及沟通效率。 |
| 平台管理员 | 负责审核认证、处理举报申诉、维护规则、统计运营数据与保障平台秩序。 |
| 志愿管理员 | 在主管理员授权下参与内容巡检、过期信息清理和基础秩序维护。 |

### 2.3 约束

1. 平台主要服务对象为高校学生，需依赖校园身份认证或校内身份校验机制。
2. 平台内容、交易与沟通行为必须符合国家法律法规及学校管理规范。
3. 系统需对虚假信息、骚扰、诈骗、失联、恶意放鸽子和不当交易具备基本治理能力。
4. 部分需求可能涉及隐私与安全风险，因此系统必须控制信息展示范围，并保留必要留痕。

### 2.4 假设

1. 用户具备手机号、学号、校园邮箱或学校统一身份认证条件中的至少一种。
2. 学校允许在合法合规范围内开展校内互助服务与信息协作。
3. 系统可接入数据库、对象存储、消息推送与日志服务等基础设施。
4. 若后续接入支付、地图、统一认证、短信服务等第三方能力，则依赖其可用性与合规性。

### 2.5 关键用例列表

| 用例编号 | 用例名称 |
|---|---|
| UC-01 | 用户注册并完成校内身份认证 |
| UC-02 | 发布互助任务/信息 |
| UC-03 | 浏览、搜索和筛选需求 |
| UC-04 | 对需求接单或报名匹配 |
| UC-05 | 在站内与对方沟通并确认细节 |
| UC-06 | 提交任务完成凭证并完成确认 |
| UC-07 | 双方互评并更新信用 |
| UC-08 | 举报违规内容或发起申诉 |
| UC-09 | 管理员/志愿管理员处理审核与治理 |
| UC-10 | 查看个人历史记录、通知与统计 |

## 附录 A 修订说明

本版本文档以 AI 草案为基础框架，以真人问卷中反映的高频需求与真实顾虑作为修订依据。与初版相比，重点增强了以下内容：

1. 补充了失物招领、资料分享、拼单拼车、找搭子/组队等更贴近真实校园场景的功能分类；
2. 将单一学号认证调整为多种校内认证方式，并加入前台匿名展示与后台可追溯并存的要求；
3. 强化了校区筛选、标签检索、过期信息清理、任务完成凭证、自动确认、志愿管理员等具体机制；
4. 在报酬与激励方面允许免费、积分、现金线下结算或双方协商，不将积分作为唯一模式。
