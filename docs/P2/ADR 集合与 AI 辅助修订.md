**ADR记录**

### **ADR-001：采用模块化单体架构 (Modular Monolith)**

* **状态**：已接受  
* **背景**：团队仅 4 人，需在 10 周内交付。微服务虽然扩展性好，但涉及服务治理和分布式事务，对学生团队成本过高 。  
* **决策**：放弃微服务，采用模块化单体架构，在单体应用内按业务领域（用户、订单、交易、资源）进行包隔离 。  
* **理由**：降低了部署和联调复杂度，确保 240 人时内能闭环业务逻辑 。  
* **后果**：  
  * 正面：开发速度极快，本地联调无障碍 。  
  * 风险：需严格执行代码审查，防止模块间产生非法耦合。  
* **AI 辅助记录**：  
  * **AI 初稿摘要**：建议使用微服务架构以保证高可用性和技术先进性 。  
  * **人工修订内容**：强制修正为“模块化单体”，移除所有关于分布式服务治理的内容 。  
  * **修订理由**：AI 忽略了真实的人力资源限制，盲目追求高级架构会导致项目烂尾 。

### **ADR-002：使用状态机模式管理订单流转**

* **状态**：已接受  
* **背景**：互助平台涉及接单、履约、确认、评价等复杂状态切换，硬编码 if-else 会导致逻辑碎片化且难以测试 。  
* **决策**：引入状态机模式（State Machine Pattern），通过事件驱动触发状态变更 。  
* **理由**：保证订单状态流转的一致性，防止资损和流程纠纷 。  
* **后果**：  
  * 正面：逻辑高度内聚，状态流转图可清晰映射到代码。  
  * 负面：初期编码工作量略有增加。  
* **AI 辅助记录**：  
  * **AI 初稿摘要**：建议在 Service 层通过简单字段更新处理状态切换。  
  * **人工修订内容**：改为“独立状态机引擎”，并要求状态变更必须发布异步事件 。  
  * **修订理由**：简单的字段更新无法应对复杂的逆向流程（如取消、纠纷），需要更健壮的模式 。

### **ADR-003：利用 Redis 实现抢单并发控制**

* **状态**：已接受  
* **背景**：校园代取快递等场景存在明显的高峰流量，多用户同时抢单的时候易引发超卖或重复接单 。  
* **决策**：在 Service 层引入 Redis 分布式锁，对订单接单接口进行限流与排队处理 。  
* **理由**：利用缓存层的高性能（QPS 瞬间冲高）保护底层 MySQL 数据库 。  
* **后果**：  
  * 正面：防止恶意刷单和并发冲突，提升高峰期系统稳定性 。  
  * 负面：系统增加了对 Redis 的依赖。  
* **AI 辅助记录**：  
  * **AI 初稿摘要**：建议使用数据库行锁 (SELECT FOR UPDATE) 处理并发。  
  * **人工修订内容**：改为 Redis 分布式锁 \+ 接口限流策略 。  
  * **修订理由**：数据库行锁在高并发下会导致连接池耗尽，不适合校园瞬间爆发的场景 。

### **ADR-004：基于 JWT 的无状态身份认证**

* **状态**：已接受  
* **背景**：项目采用前后端分离架构（Vue 3 \+ SpringBoot），需要一种轻量、易分工的鉴权方式 。  
* **决策**：采用 JWT (JSON Web Token) 进行身份验证，客户端存储 Token，服务端不保存 Session 状态。  
* **理由**：符合前后端分离的文档要求，且方便后续接入 CI 自动化测试 。  
* **后果**：  
  * 正面：服务端无状态，易于水平扩展。  
  * 风险：Token 签发后难以撤回，需配合较短的过期时间。  
* **AI 辅助记录**：  
  * **AI 初稿摘要**：推荐使用传统的 Session 或 OAuth2 复杂流程。  
  * **人工修订内容**：简化为“JWT \+ 拦截器”模式 。  
  * **修订理由**：OAuth2 对于中小型课程项目过于沉重，JWT 在 10 周开发周期内更具可行性 。

### **ADR-005：基础设施强制容器化 (Docker Compose)**

* **状态**：已接受  
* **背景**：团队成员技术背景和系统环境不一，数据库版本冲突曾影响 P1 实验进度 。  
* **决策**：强制所有中间件（MySQL, Redis）必须通过 Docker Compose 一键拉起 。  
* **理由**：确保“环境即代码”，消除“我机器上能跑”的经典协作问题 。  
* **后果**：  
  * 正面：新成员克隆仓库后可分钟级搭建好开发环境 。  
  * 风险：需要成员掌握基础的 Docker 操作。  
* **AI 辅助记录**：  
  * **AI 初稿摘要**：建议在本地安装 MySQL 8.0 社区版并手动配置。  
  * **人工修订内容**：改为“强制 Docker Compose 部署”，并配置持久化挂载 。  
  * **修订理由**：手动配置极易因版本偏差（如字符集、时区）导致联调失败，容器化是团队分工的最佳保障 。
