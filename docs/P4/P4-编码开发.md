# Phase 4：编码开发

**周次：** 第 10-14 周 | **人均工时：** 10 小时 | **团队总工时：** 40 小时 | **提交时间：** 2026 年 6 月 7 日

---

## 一、本阶段目标

1. 实现 P0 优先级核心功能，并完成 P1 优先级功能的基本开发
2. 完成前后端联调与模块集成，形成可运行、可演示的系统版本
3. 编写单元测试与集成测试，覆盖核心业务逻辑和主要业务流程
4. 通过"AI 代码信任度实验"与"AI 调试对决"，分析 AI 在编码和调试中的价值与局限
5. 准备演示环境

---

## 二、任务说明

### 任务 1：Sprint 合并规划（4 小时）

#### 1.1 任务拆解

将 P0 与 P1 优先级功能拆分为具体开发任务。每个任务应：
- 可在 2-4 小时内完成
- 有明确的完成标准
- 分配给具体负责人
- 标明优先级（P0 / P1 / 可选 P2）

拆解说明：
- 本阶段优先保证 P0 功能形成可演示闭环：用户登录注册 -> 发布需求 -> 浏览详情 -> 接单 -> 状态流转 -> 查看订单历史。
- P1 功能以“基本可用”为目标，评价信用和通知只实现课程验收所需的核心规则，不做复杂推荐、实时聊天和外部推送。
- P2 功能只在 P0/P1 稳定后开发，智能匹配和管理后台均采用轻量实现，不能影响主流程稳定性。
- 每个任务使用独立 `feature/*` 分支开发，命名遵循 `BRANCH_STRATEGY.md`，合并到 `develop` 前至少完成自测和 1 人代码确认。

#### 1.2 AI 辅助 Sprint 规划

可以让 AI 辅助进行任务拆解、工时估算和风险识别，但团队需要人工确认：
- 哪些功能必须进入本阶段交付
- 哪些功能可以降级或延期
- 哪些任务存在技术风险或集成风险
- AI 给出的工时估算是否符合团队实际能力

### 任务 2：核心功能与剩余功能开发（20 小时）

#### 2.1 P0 核心功能开发

完成系统最小可用版本所需的核心功能。具体范围以 P1 需求分析和 P3 详细设计为准，通常包括：
- 用户注册、登录与身份认证
- 需求发布、列表查询、详情查看
- 接单、订单状态流转、订单历史
- 数据库初始化、基础接口、统一错误处理等支撑能力

#### 2.2 P1 功能完善

完成 P1 优先级功能的基本开发：

| 功能 | 关键点 | AI 可辅助 | 需人工把控 |
|------|-------|---------|----------|
| 评价与信用体系 | 双向评分逻辑、信用分计算、评价记录 | 基础 CRUD、测试样例生成 | 信用分算法的公平性与可解释性 |
| 消息通知 | 通知触发时机、消息模板、通知状态 | 消息模板、接口样板代码 | 通知策略，避免重复通知或过度打扰 |

如果时间允许，可选做 P2 功能，如智能匹配、管理后台、数据统计等。可选功能不应影响 P0/P1 功能的稳定性。

### 任务 3：关键实验一 —— "AI 代码信任度实验"（4 小时）

选择一个规模适中的功能点，建议选择**登录校验**、**订单状态更新**、**评价提交**、**消息通知**等，执行一个简化实验。重点不是做复杂评测，而是观察：AI 生成的代码能不能直接用？人工检查发现了什么问题？

#### 步骤一：AI 直接生成

1. 用一个清晰 Prompt，让 AI 生成该功能点的代码或核心逻辑
2. 保留 AI 原始输出，记录 Prompt 和生成结果摘要
3. 将代码放入项目中运行，记录能否编译、能否运行、是否通过已有测试

#### 步骤二：人工检查与修复

1. 人工阅读 AI 生成代码，至少检查以下 3 点：
   - 是否符合 P3 详细设计
   - 是否处理了常见异常或边界情况
   - 是否存在明显 Bug 或不合理逻辑
2. 记录发现的 1-3 个主要问题；如果没有发现问题，也要说明检查依据
3. 修复问题后重新运行项目或测试

#### 步骤三：记录实验

| 指标 | AI 直出 | 人工审查修复后 |
|------|--------|-------------|
| 编译是否通过 | | |
| 功能是否可运行 | | |
| 测试是否通过 | | |
| 主要问题/修复说明 | | |

### 任务 4：单元测试、集成测试与 CI/CD（4 小时）

#### 4.1 单元测试

核心业务逻辑必须有单元测试，重点覆盖：
- 边界条件
- 异常输入
- 权限校验
- 状态转换
- 信用分、订单状态等关键业务规则

目标：核心模块单元测试覆盖率 >= 60%

#### 4.2 AI 辅助测试审查

可以让 AI 生成测试用例，但必须人工审查：
- 是否覆盖边界条件
- 是否覆盖异常路径
- 是否存在"看起来测试了，实际没有断言关键结果"的空洞测试
- 是否遗漏重要业务场景

#### 4.3 集成测试

编写集成测试，至少覆盖：
- 1 个完整正常流程：用户注册 -> 登录 -> 发布需求 -> 另一用户接单 -> 完成 -> 评价
- 2 个异常流程：如未登录访问、重复接单、无效参数、越权操作等
- 前后端联调中的关键接口调用

#### 4.4 CI/CD 配置

配置基础 CI/CD 流程，确保代码提交后能够自动完成质量检查，建议使用GitLab CI/CD。

CI/CD 至少应包括：
- 自动安装依赖
- 自动运行静态检查或代码格式检查
- 自动运行单元测试
- 自动运行集成测试
- 自动构建项目，验证项目可以正常打包、部署

### 任务 5：模块集成与 AI 调试对决（4 小时）

#### 5.1 系统联调

完成前后端和各模块之间的集成，重点检查：
- 接口字段是否一致
- 认证状态是否正确传递
- 订单、评价、通知等跨模块流程是否连贯
- 异常提示是否清晰
- 数据库状态是否符合业务预期

#### 5.2 关键实验二 —— "AI 调试对决"

团队在集成过程中选择 2-3 个有代表性的 Bug，比较纯人工调试和 AI 辅助调试的差异。

每个 Bug 的处理流程：

| 步骤 | 纯人工调试 | AI 辅助调试 |
|------|----------|-----------|
| 1. 问题描述 | 团队成员自行分析报错信息 | 将报错信息、复现步骤和相关代码提供给 AI |
| 2. 定位耗时 | 记录从开始到定位根因的时间 | 记录 AI 给出定位建议的时间 |
| 3. 定位准确度 | 记录人工判断过程 | 判断 AI 的定位是否正确 |
| 4. 修复方案 | 人工设计修复方案 | 记录 AI 建议的修复方案 |
| 5. 方案质量 | 记录人工方案的优缺点 | 判断 AI 方案是否引入新问题 |
| 6. 最终修复 | 记录最终采用的方案和原因 | 说明是否采纳 AI 建议 |

实验记录表：

| Bug # | Bug 描述 | 纯人工定位耗时 | AI 辅助定位耗时 | AI 定位是否准确 | AI 修复方案是否可用 | 最终方案来源 |
|-------|---------|-------------|-------------|-------------|-----------------|-----------|
| 1 | | | | | | |
| 2 | | | | | | |

分析问题：
- AI 在哪类 Bug 上表现较好？
- AI 在哪类 Bug 上表现较差？
- AI 是否出现只修表象、不找根因的情况？
- 人工提供哪些上下文后，AI 的调试建议明显变好？

### 任务 6：演示准备与版本收尾（4 小时）

完成本阶段可演示版本的准备：
1. 准备本地运行或简单部署环境
2. 准备核心功能演示路径
3. 修复影响演示的关键 Bug
4. 整理 Bug 修复日志，记录问题现象、根因、修复方案和验证结果
5. 更新 README 或运行说明，确保助教可以体验项目
